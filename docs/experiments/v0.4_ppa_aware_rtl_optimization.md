# v0.4 PPA-Aware RTL Optimization Experiment

## Purpose

v0.4 designs a correctness-gated scoring layer for RTL optimization before any new model jobs are run.

The research question is:

> Can `qwen36-27b` generate RTL that remains functionally equivalent to a reference design while improving area and/or power under controlled synthesis assumptions?

This is an optimization experiment, not a general RTL-generation leaderboard. The initial public validation target is RTL-OPT. SiliconCraft/RFID private tasks come later, only after the public scoring flow is validated and reproducible.

## Relationship To v0.3

v0.3 compared prompt profiles using functional outcomes. It established that:

- `neutral_baseline` remains the default profile.
- `strict_rtl_only` remains an optional challenger.
- `low_area_low_power` remains held back.
- `no_explanation_code_only` remains held back.

Those results cannot support area or power claims. A prompt can request fewer registers, shared resources, or lower switching without actually improving a synthesized implementation. Functional pass rates also do not measure physical cost. v0.4 therefore requires measured synthesis data and functional equivalence before awarding any optimization credit.

## Initial Scope

The first implementation stage should use:

- benchmark family: public RTL-OPT
- model: `qwen36-27b`
- initial control profile: `neutral_baseline`
- challenger profiles: `strict_rtl_only` and, after flow validation, `low_area_low_power`
- primary objectives: area and power
- timing treatment: constraint/gate, not reward

No private SiliconCraft/RFID task should enter the flow until public RTL-OPT extraction, synthesis, equivalence, metric capture, scoring, sanitization, and reporting have been validated end to end.

## Correctness Gates

A generated sample is eligible for an optimization score only after the required gates pass in order:

1. extraction succeeds
2. syntax/compile succeeds
3. synthesis succeeds
4. equivalence succeeds
5. timing constraint succeeds when timing data and a declared constraint are available

The pipeline should stop PPA scoring at the first failed required gate. A smaller design that is incorrect, unsynthesizable, inequivalent, or timing-invalid is not an optimization success.

Gate outcomes should be recorded separately so reports can distinguish generation failures from PPA regressions.

## Scoring Model

When area and power are both available and all required gates pass:

```text
score = 0.5 * area_score + 0.5 * power_score
area_score = min(reference_area / generated_area, 2.0)
power_score = min(reference_power / generated_power, 2.0)
```

Interpretation:

- `1.0` means parity with the reference for that metric.
- greater than `1.0` means improvement.
- less than `1.0` means regression.
- each component is capped at `2.0` to prevent a single extreme ratio from dominating aggregate results.

Reference and generated metrics must come from the same synthesis/power flow, technology assumptions, constraints, tool versions, and reporting units.

## Timing Treatment

Timing is a constraint, not a reward in the first scoring version.

When timing is available, record the declared constraint, achieved timing or slack, and `timing_pass`. A timing failure makes the optimization score invalid or zero according to the reporting context. Positive slack does not add bonus points to the area/power score.

When timing is unavailable, record `timing_status: unavailable`. Do not silently treat unavailable timing as a measured pass. Early public-flow reports must clearly state whether timing was enforced.

## Missing Public Power Data

If the public RTL-OPT flow cannot provide trustworthy power measurements, start in explicit area-only mode:

- compute `area_score`
- set `power_status: power_unavailable`
- leave `reference_power`, `generated_power`, and `power_score` null
- set `scoring_mode: area_only`
- use `optimization_score = area_score`

Area-only results must be labeled as area-only and must not be described as PPA or low-power improvements. Do not infer power from area, cell count, RTL style, or prompt wording.

## Invalid And Zero Scores

Two representations are useful and should remain distinguishable:

- Per-sample analytical artifacts: failed required gates use `score_status: invalid` and a null `optimization_score`.
- Aggregate ranking views: invalid samples contribute zero optimization credit so failure rates cannot disappear from averages.

Reports must publish both valid-score statistics and all-sample zero-filled statistics, together with the count of invalid samples. This prevents a profile with very few correct samples from looking strong by averaging only its survivors.

## Planned Sanitized Artifacts

Future export and summary helpers may be named:

- `scripts/export_v0.4_ppa_results.py`
- `scripts/summarize_v0.4_ppa_results.py`

Sanitized per-sample rows should contain only structured metadata and measured values. The proposed schema is defined in `reports/v0.4_ppa_scoring_design.md`.

The export must exclude raw RTL, prompts, model responses, error-log contents, credentials, private task text, and paths that expose those artifacts.

## Retesting `low_area_low_power`

Retest `low_area_low_power` only after the public scoring implementation passes synthetic/unit tests and a small control-flow validation.

The later comparison should use matched:

- RTL-OPT task set
- model and endpoint
- generation settings and sample count
- synthesis tools, versions, libraries, constraints, and seeds
- equivalence and timing rules
- scoring mode

Compare at least `neutral_baseline` against `low_area_low_power`; retain `strict_rtl_only` only if the research question includes formatting effects. Report correctness-gate pass rates before optimization scores. Promotion requires equivalent designs and measured area/power evidence, not prompt intent.

## Public RTL-OPT Limitations

The current public flow is useful for validating synthesis and equivalence mechanics, but it does not automatically provide a complete physical-design PPA result.

Important limitations include:

- power may be unavailable or based on insufficient activity information
- timing constraints and timing reports may be absent or incomplete
- synthesis area is tool/library dependent and is not final placed-and-routed silicon area
- cell count and generic synthesis statistics are proxies, not universal physical metrics
- task coverage is limited and may not represent production designs
- synthesis heuristics and tool versions can change reported values

All public reports must identify the exact available metrics and avoid extending area-only evidence into power, timing, or final-silicon claims.

## Staged Plan

1. Freeze the scoring contract and sanitized schema.
2. Add pure scoring functions and unit tests.
3. Add sanitized export and summary helpers.
4. Validate the public RTL-OPT flow using existing or deliberately small control data.
5. Confirm equivalence, metric units, missing-power behavior, and aggregate invalid handling.
6. Retest `low_area_low_power` against the control profile only after the public flow is trustworthy.
7. Consider SiliconCraft/RFID private tasks after public validation, under a separate private-data policy.

No benchmark execution is part of this design stage.
