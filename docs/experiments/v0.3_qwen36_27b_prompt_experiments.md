# Baseline v0.3 qwen36-27b Prompt Experiments

Baseline v0.2 is frozen before this experiment series begins. Its registered run selection, sanitized per-task artifacts, consistency audit, failure matrix, and dashboard remain reference data and must not be overwritten by v0.3 results.

## Research Question

Baseline v0.3 compares prompt strategies on qwen36-27b only. It is a prompt experiment, not a model comparison. qwen36-27b remains the fixed quality reference so changes can be attributed to prompt behavior under otherwise matched settings.

The available profiles in `configs/prompt_profiles.yaml` are:

- `neutral_baseline`: preserves the historical system prompt.
- `strict_rtl_only`: requests synthesizable RTL with no prose, Markdown, or unsolicited testbench.
- `low_area_low_power`: prioritizes area and power while treating timing as a constraint, without making unsupported PPA claims.
- `no_explanation_code_only`: applies the strictest output formatting for code-extraction reliability.

When `--prompt-profile` and `prompt.profile` are both absent, RTLBench uses the unchanged historical system prompt. Selecting `neutral_baseline` makes that same prompt explicit in the run metadata.

## Smoke First

Start with the three-task configs under `configs/experiments/`:

```bash
.conda-env/bin/rtlbench \
  --config configs/experiments/v0.3_qwen36_27b_verilogeval_prompt_smoke.yaml \
  --base-url "$OPENAI_BASE_URL" \
  --prompt-profile strict_rtl_only \
  --output-dir outputs/experiments/v0.3_prompt_profiles/verilogeval/strict_rtl_only \
  --notes "v0.3 strict_rtl_only VerilogEval smoke"
```

Equivalent smoke configs exist for RTLLM 2.0 and RTL-OPT equivalence. Do not expand to full task sets until the smoke outputs have valid API responses, RTL extraction, evaluator execution, and metadata.

## Run Isolation

Use a distinct base output directory for every benchmark and prompt profile:

```text
outputs/experiments/v0.3_prompt_profiles/<benchmark>/<prompt_profile>/<benchmark>/<model>/<timestamp>/
```

The final benchmark/model/timestamp components are added by the existing runner. This keeps current output behavior backward compatible. Never write experiment results into baseline v0.1 or v0.2 artifact/report paths.

## Comparison Rules

Keep the benchmark version, task subset, qwen36-27b endpoint, temperature, top-p, max tokens, samples per task, timeout, and evaluator identical across prompt profiles. Record the selected profile in `config_snapshot.yaml`, `run_metadata.json`, and `report.md`.

Reuse the existing pass/fail metrics, sanitized per-task export, cross-model failure analysis machinery, and dashboard flow in experiment-specific paths. Treat `low_area_low_power` as a prompt hypothesis only: exact PPA claims require synthesis reports and functional equivalence evidence.
