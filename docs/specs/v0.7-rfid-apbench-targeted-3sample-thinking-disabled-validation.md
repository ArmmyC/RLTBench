# Feature Spec: v0.7 RFID-APBench Targeted 3-Sample Thinking-Disabled Validation

## Goal

Confirm whether the post-bugfix thinking-disabled RFID runner reliably avoids the prior null-content token-limit symptom on the two targeted public/synthetic tasks.

The bounded one-sample validation showed both targeted tasks returned visible content with `finish_reason=stop` after `chat_template_kwargs.enable_thinking=false` was forwarded. Because that validation was only one sample per task, this follow-up should run the standard 3-sample-per-task targeted confirmation before considering max-token diagnostics, broader reruns, or any training-related work.

This is a bounded targeted validation step only. It is not a benchmark-wide baseline, benchmark modification, prompt tuning, model comparison, private evaluation, fine-tuning, or training-data creation.

Read first:

- `reports/v0.7_rfid_apbench_bounded_thinking_disabled_validation.md`
- `reports/v0.7_rfid_apbench_bounded_thinking_disabled_validation.csv`
- `reports/v0.7_rfid_apbench_bounded_thinking_disabled_validation.jsonl`
- `reports/v0.7_rfid_apbench_thinking_mode_configuration_bugfix.md`
- `reports/v0.7_rfid_apbench_endpoint_runtime_investigation.md`
- `reports/v0.7_rfid_apbench_profile_rejected_closeout.md`
- `configs/models.yaml`
- `configs/prompt_profiles.yaml`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `src/rtlbench/client.py`

## Required Scope

Run a targeted validation limited to:

- tasks: `ap_001_idle_counter`, `ap_010_retry_timeout_fsm`
- samples: exactly 3 samples per task, 6 attempted samples total
- profile: `neutral_baseline`
- model: configured/default `qwen36-27b` when available
- max tokens: unchanged `4096`
- temperature/top-p: unchanged deterministic defaults
- benchmark assets: unchanged
- extractor/evaluator/scoring: unchanged
- committed outputs: sanitized reports only

The validation must use the fixed RFID runner path so requests include the configured model preset `extra_body`, especially `chat_template_kwargs.enable_thinking=false`.

If endpoint or tools are unavailable, do not fake results. Emit sanitized blocker/failure rows or a sanitized note.

## Required Files

Create:

- `reports/v0.7_rfid_apbench_targeted_3sample_thinking_disabled_validation.md`
- `reports/v0.7_rfid_apbench_targeted_3sample_thinking_disabled_validation.csv`
- `reports/v0.7_rfid_apbench_targeted_3sample_thinking_disabled_validation.jsonl`

Do not overwrite historical reports.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- validation configuration
- confirmation that the fixed path includes the thinking-disable extra body
- endpoint and tool status without secrets
- total attempted samples
- request outcome distribution
- per-task empty/non-empty rates
- per-sample sanitized request outcome summary
- content presence, character count, finish reason, completion-token count, parse status, and HTTP class
- extraction/evaluation gate summary
- valid-score summary and all-sample zero-filled score
- comparison against pre-bugfix targeted evidence and the one-sample post-bugfix validation
- interpretation of whether the null-content token-limit symptom is resolved, reduced, persistent, or inconclusive on these two tasks
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next step
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Decision Rules

Recommend exactly one:

- `v0.7_thinking_disabled_closeout_spec` if no null-content token-limit rows recur and results are interpretable.
- `bounded_max_tokens_diagnostic_spec` if null-content token-limit behavior persists even with thinking disabled and a max-token diagnostic is now justified.
- `serving_stack_issue_spec` if null-content behavior persists and evidence points to endpoint/server behavior outside RTLBench configuration.
- `client_response_capture_observability_spec` if sanitized metadata is still insufficient to interpret the response boundary.
- `no_immediate_followup_beyond_documentation` if validation cannot be performed or yields no useful evidence.

Do not recommend fine-tuning, private evaluation, model comparison, broad benchmark rerun, prompt-profile experiment, benchmark modification, or training-data creation.

## Definition of Done

Done when:

1. Sanitized Markdown/CSV/JSONL targeted validation reports exist.
2. The validation attempts exactly 3 samples each for `ap_001_idle_counter` and `ap_010_retry_timeout_fsm`, or explicitly reports endpoint/tool blockers.
3. It confirms the fixed runner path includes the thinking-disable extra body.
4. It reports request/content/token metadata without raw outputs.
5. It states whether the null-content token-limit symptom is resolved, reduced, persistent, or inconclusive.
6. Benchmark assets remain unchanged.
7. Fine-tuning is not recommended from this evidence alone.
8. Exactly one next bounded step is recommended.
9. `python -m pytest` passes.
10. No raw/private/model-output/training artifacts are committed.
