# Feature Spec: v0.8 RFID-APBench Report Hygiene

## Goal

Define stable RFID-APBench report hygiene expectations before adding automated CI/release assertions.

The v0.8 reproducibility documentation recommends `v0.8_report_hygiene_spec` because the release checklist and reproducibility guide now cover pre-tag gates and user-facing rerun instructions. The next hardening step is to define stable report schema, lineage, naming, sanitized-field expectations, and caveat requirements for committed RFID-APBench evidence.

This is documentation/report-hygiene hardening only. It is not a model evaluation, benchmark modification, prompt-profile experiment, model comparison, private evaluation, fine-tuning, or training-data creation.

## Required Scope

Create a report-hygiene artifact for RFID-APBench committed evidence. It must define expectations for:

1. Stable report naming: version prefix, benchmark name, scope, and artifact kind.
2. Required Markdown sections for baseline reports, targeted validations, failure analyses, release closeouts, and planning reports.
3. CSV/JSONL row schema expectations for runs that produce per-sample rows.
4. Request and response metadata fields allowed in sanitized reports.
5. Extraction/evaluation gate fields: extraction, compile, functional simulation, synthesis, timing status, area metric, activity metric, validity, and failure category.
6. Score field expectations: valid-score count, mean valid score, all-sample zero-filled score, area/activity score terms, and invalid-row zero-fill behavior.
7. Source-evidence and lineage requirements.
8. Required caveats: activity proxy, public/synthetic scope, no private evaluation, not fine-tuning readiness, no measured-power/final-PPA claims.
9. Raw/prohibited artifact exclusions.
10. Known-limitation and benchmark-asset decision expectations.
11. Suggested checklist for reviewing new report artifacts before commit.

Do not run a model evaluation in this spec.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- summary of report-hygiene document created
- naming convention summary
- required section summary
- sanitized row schema summary
- prohibited artifact summary
- how release reviewers should use the hygiene document with the release checklist and reproducibility docs
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next bounded direction
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Report Hygiene Document Requirements

`docs/release/rfid_apbench_report_hygiene.md` must be practical and review-facing. It should include:

- report filename conventions
- required Markdown sections by report type
- CSV/JSONL required and optional fields
- allowed sanitized endpoint/request/token metadata
- required gate and score fields
- failure category guidance
- lineage/source-evidence guidance
- caveat and claim-safety guidance
- prohibited raw/private artifact checklist
- example review checklist for a new report
- guidance for when no CSV/JSONL is required

## Decision Rules

Recommend exactly one:

- `v0.8_ci_release_assertions_spec` if report hygiene is complete and automated release/report checks should happen next.
- `no_immediate_followup_beyond_documentation` if the hygiene doc is sufficient for now.

Do not recommend fine-tuning, private evaluation, model comparison, global max-token changes, prompt-profile experiments, benchmark modification, broad rerun, or training-data creation from this documentation step.

## Tests

Run:

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompt text
- raw model responses
- response bodies
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- run another model evaluation
- add or modify benchmark tasks
- edit prompt text
- edit reference RTL or testbench behavior
- change module interfaces
- compare models
- change prompt profiles
- raise global max-token default
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change workload, metrics, area/activity scoring, or evaluator policy
- create or push git tags

## Definition of Done

Done when:

1. `docs/release/rfid_apbench_report_hygiene.md` exists.
2. `reports/v0.8_rfid_apbench_report_hygiene.md` exists.
3. The hygiene doc defines naming, sections, row schema, sanitized metadata, gates, scores, lineage, caveats, artifact exclusions, and review checklist expectations.
4. The report explains how to use the hygiene doc with release checklist and reproducibility docs.
5. Benchmark assets remain unchanged.
6. Fine-tuning is not recommended from this evidence alone.
7. Exactly one next bounded direction is recommended.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
