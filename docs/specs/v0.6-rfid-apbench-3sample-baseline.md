# Feature Spec: v0.6 RFID-APBench 3-Sample Baseline

## Goal

Start v0.6 by running a bounded 3-sample baseline on the existing five public/synthetic RFID-APBench tasks using the same benchmark assets and scoring path from v0.5.

This stage should turn the one-sample v0.5 model smoke into a more useful repeated-sample baseline without adding new tasks, changing prompts, changing references, tuning prompts, or starting fine-tuning.

## Inputs

Read first:

- `reports/v0.5_rfid_apbench_closeout.md`
- `docs/specs/v0.5-rfid-apbench-closeout-and-v0.6-roadmap.md`
- `reports/v0.5_rfid_apbench_model_smoke.md`
- `reports/v0.5_rfid_apbench_model_smoke_analysis.md`
- `reports/v0.5_rfid_apbench_idle_counter_retry.md`
- `reports/v0.5_rfid_apbench_idle_counter_correctness_investigation.md`
- `scripts/run_rfid_apbench_model_smoke.py`
- `scripts/evaluate_rfid_apbench_candidates.py`

## Required Scope

Implement or run a bounded v0.6 baseline path that:

1. Uses the existing five RFID-APBench tasks only.
2. Runs exactly 3 samples per task.
3. Uses the same public/synthetic prompts, references, testbenches, activity workloads, and reference metrics.
4. Uses the same configured/default model as v0.5, expected `qwen36-27b` when available.
5. Uses `neutral_baseline` or the current default RTLBench prompt behavior.
6. Uses low/deterministic generation settings where supported.
7. Extracts generated RTL into ignored output/work directories only.
8. Evaluates each extracted candidate through compile, simulation correctness, VCD activity, synthesis, reference metrics, and `area_activity` scoring.
9. Emits sanitized Markdown/CSV/JSONL reports.
10. Keeps raw prompts, raw model responses, generated RTL, VCDs, logs, compiled artifacts, and synthesis scratch out of committed paths.

## Required Files

Create or update as appropriate:

- `scripts/run_rfid_apbench_3sample_baseline.py` or a compatible CLI option on an existing script
- `reports/v0.6_rfid_apbench_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_3sample_baseline.csv`
- `reports/v0.6_rfid_apbench_3sample_baseline.jsonl`
- optional tests for pure helper behavior if new helper code is introduced

Do not commit raw run directories or generated RTL.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- model and endpoint/tool availability without secrets
- task count and samples per task
- total sample count
- generation completion counts
- extraction pass counts
- candidate file availability counts
- compile/correctness/synthesis counts
- area/activity metric availability counts
- valid score count
- mean valid score
- mean all-sample score with invalid rows as zero
- per-task aggregate summary
- per-sample sanitized summary
- failure category distribution
- score distribution notes
- comparison against v0.5 one-sample smoke where useful
- caveat that activity is VCD toggle-count proxy, not measured power
- caveat that this is public synthetic evaluation, not private evaluation and not fine-tuning

CSV/JSONL rows must include sanitized per-sample fields such as:

- benchmark
- task_id
- sample_id
- model
- prompt_profile
- generation settings
- generation_status
- extraction_status
- candidate_file_available
- compile_pass
- correctness_pass
- synth_pass
- timing_status
- reference_area
- generated_area
- reference_activity
- generated_activity
- area_score
- activity_score
- score
- score_status
- failure_category
- toolchain_id
- workload_id
- sanitized notes

## Failure Taxonomy

At minimum, classify failures into these sanitized categories when applicable:

- endpoint_unavailable
- request_failed
- empty_response
- incomplete_rtl
- extraction_failure
- candidate_missing
- interface_or_top_module_mismatch
- compile_failure
- reset_behavior_mismatch
- clear_behavior_mismatch
- terminal_count_mismatch
- idle_hold_mismatch
- state_update_timing_mismatch
- simulation_failure
- synthesis_failure
- area_metric_unavailable
- activity_metric_unavailable
- score_regression
- passed

Do not include raw generated RTL, raw model response text, raw prompts, simulator log excerpts, VCD snippets, or raw artifact paths in committed reports.

## Operational Blockers

If the model endpoint or EDA tools are unavailable, write a clear sanitized blocker report instead of faking outputs.

Do not change benchmark conclusions from an operational blocker.

## Tests

Run:

```bash
python -m pytest
```

Add tests only for pure helpers if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- model outputs
- raw benchmark run outputs
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- add new benchmark tasks
- modify existing tasks, prompts, references, testbenches, extractor, evaluator, or scoring policy
- compare multiple models
- tune prompts broadly
- run more than 3 samples per task
- create training data
- fine-tune a model
- claim measured power
- claim final silicon PPA
- integrate private benchmarks

## Definition of Done

Done when:

1. The existing five tasks each have exactly 3 attempted samples.
2. Sanitized Markdown/CSV/JSONL baseline reports exist.
3. The reports include per-task and per-sample summaries.
4. Failure categories are recorded without raw artifacts.
5. Valid and all-sample scores are reported.
6. `python -m pytest` passes.
7. No raw/private/model-output/training artifacts are committed.
