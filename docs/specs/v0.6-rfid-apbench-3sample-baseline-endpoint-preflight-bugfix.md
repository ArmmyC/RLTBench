# Bugfix Spec: v0.6 RFID-APBench 3-Sample Baseline Endpoint Preflight

## Goal

Fix the v0.6 3-sample baseline runner so a configured OpenAI-compatible endpoint is not treated as unavailable solely because `/models` does not answer.

The current baseline report produced 15 `endpoint_unavailable` rows with the note `configured endpoint did not answer model listing`, even though earlier v0.5 smoke workflows generated through the same class of local OpenAI-compatible endpoint. The baseline should attempt the bounded generation path when base URL and credential are configured, and record per-sample request failures if generation itself fails.

This is a bugfix only. It must not change benchmark tasks, prompts, references, extractor, evaluator, scoring policy, or sample bounds.

## Implementation Status

Status after review: pending implementation.

`main` currently has no implementation commit after this bugfix spec. The runner still needs to be changed so `/models` listing failure is not a hard blocker by default.

## Problem

`run_rfid_apbench_3sample_baseline.py` currently checks `/models` with `endpoint_reachable()` before generation. If this preflight fails, it writes 15 endpoint-blocker rows and never attempts generation.

This can be too strict for local OpenAI-compatible endpoints that support chat/completions but do not reliably expose `/models`, or where `/models` is temporarily unavailable while generation would still return a useful per-request error.

## Required Scope

Update the baseline runner so that:

1. Missing base URL or credential still produces a sanitized `endpoint_unavailable` blocker report.
2. Missing Icarus/vvp/Yosys still produces a sanitized `tool_unavailable` blocker report.
3. A failed `/models` listing check does not automatically block generation by default.
4. The script attempts the existing bounded 5-task x 3-sample generation path when endpoint config and EDA tools are present.
5. If generation requests fail, each failed sample is recorded as `request_failed` with sanitized notes.
6. Optionally keep `/models` as a non-blocking diagnostic in the report, or add an explicit opt-in flag such as `--require-models-preflight` if a hard preflight is desired.
7. Reports remain sanitized and must not include secrets, raw prompts, raw responses, generated RTL, logs, VCDs, or raw artifact paths.

## Expected Behavior

### Config missing

If base URL or credential is missing:

- produce 15 sanitized `endpoint_unavailable` rows
- do not fake generation

### Tools missing

If EDA tools are missing:

- produce 15 sanitized `tool_unavailable` rows
- do not fake evaluation

### `/models` unavailable but config present

By default:

- do not block immediately
- attempt generation for the 15 samples
- record actual `completed`, `request_failed`, `extraction_failure`, `simulation_failure`, or `passed` outcomes

### Generation unavailable

If all generation requests fail:

- produce 15 sanitized `request_failed` rows
- report that endpoint configuration exists but generation requests failed
- do not call it a missing endpoint unless base URL/credential are missing

## Definition of Done

Done when:

1. The baseline runner no longer hard-blocks on `/models` by default.
2. Missing config and missing tools still produce clear blocker reports.
3. Failed generation requests become sanitized per-sample `request_failed` rows.
4. Baseline reports are regenerated or explicitly left as current blocker reports with an explanation if no endpoint is available.
5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
