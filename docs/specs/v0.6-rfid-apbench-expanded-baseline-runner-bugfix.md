# Bugfix Spec: v0.6 RFID-APBench Expanded Baseline Runner Task Count

## Goal

Fix the v0.6 RFID-APBench 3-sample baseline runner so it works after RFID-APBench expanded from 5 to 10 tasks.

The task expansion is complete and the adapter now loads 10 public/synthetic tasks, but `scripts/run_rfid_apbench_3sample_baseline.py` still hard-codes `EXPECTED_TASKS = 5`. This makes the runner fail before generation/evaluation on the expanded benchmark.

This is a runner compatibility bugfix only. It is not a model rerun, task expansion, prompt tuning, model comparison, private evaluation, or fine-tuning step.

## Inputs

Read first:

- `docs/specs/v0.6-rfid-apbench-task-expansion.md`
- `reports/v0.6_rfid_apbench_task_expansion_reference_validation.md`
- `reports/v0.6_rfid_apbench_task_expansion_candidate_smoke.md`
- `benchmarks/rfid_apbench/manifest.yaml`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `tests/test_rfid_3sample_baseline.py`
- `tests/test_rfid_apbench_adapter.py`

## Required Scope

Update the 3-sample baseline runner so that:

1. It no longer hard-codes the benchmark to exactly 5 tasks.
2. It loads the current task count from the RFID-APBench adapter/manifest.
3. It still enforces exactly 3 samples per task for this runner.
4. It reports the actual task count and total sample count in Markdown output.
5. Blocker rows cover all currently loaded tasks.
6. Unit tests cover the expanded 10-task case.
7. The change does not run a model baseline.
8. The change does not modify benchmark tasks, prompts, references, testbenches, extractor policy, evaluator policy, or scoring formula.

## Required Files

Update as appropriate:

- `scripts/run_rfid_apbench_3sample_baseline.py`
- `tests/test_rfid_3sample_baseline.py`
- optional documentation comments if needed

Do not regenerate model-evaluation reports in this bugfix unless no model endpoint is required and the output is a sanitized blocker-only dry run. Prefer tests over report churn.

## Expected Behavior

With the current expanded benchmark:

- `load_tasks()` should return 10 tasks without error.
- blocker rows should produce 30 rows for 10 tasks × 3 samples.
- the runner should still reject `--samples-per-task` values other than 3.
- output reports should say task count 10 and total sample count 30 when run on the expanded benchmark.

If the benchmark grows again in a future spec, the runner should not require another hard-coded task-count edit.

## Tests

Run:

```bash
python -m pytest
```

Tests should include:

- current adapter loads exactly 10 tasks
- runner `load_tasks()` accepts the current expanded benchmark
- blocker rows cover 10 tasks × 3 samples
- `--samples-per-task` remains bounded to 3
- report writer uses the actual task count and sample count

Do not require a live model endpoint in unit tests.
