# Feature Spec: v0.7 RFID-APBench ap_001 Post-Clarity Targeted Validation

## Goal

Validate whether the clarified public `ap_001_idle_counter` prompt improves candidate functional behavior under the fixed thinking-disabled RFID runner path.

The `ap_001` prompt clarity update made the existing terminal-count contract explicit: when `active` is high and `count` is `8'h0e`, the next rising clock edge must update `count` to `8'h0f` and assert `expired` in the same sequential update. Reference RTL, testbench behavior, module interface, workload, metrics, evaluator policy, scoring, model settings, and token limits remain unchanged.

This validation is narrowly scoped to `ap_001_idle_counter`. It is not a benchmark-wide baseline, model comparison, prompt-profile experiment, private evaluation, fine-tuning, or training-data creation.

Read first:

- `reports/v0.7_rfid_apbench_ap001_prompt_clarity.md`
- `reports/v0.7_rfid_apbench_ap001_correctness_investigation.md`
- `reports/v0.7_rfid_apbench_thinking_disabled_closeout.md`
- `reports/v0.7_rfid_apbench_targeted_3sample_thinking_disabled_validation.md`
- `benchmarks/rfid_apbench/tasks/ap_001_idle_counter/prompt.md`
- `benchmarks/rfid_apbench/tasks/ap_001_idle_counter/task.yaml`
- `benchmarks/rfid_apbench/tasks/ap_001_idle_counter/reference.sv`

- `scripts/run_rfid_apbench_3sample_baseline.py`
- `scripts/evaluate_rfid_apbench_candidates.py`

## Required Scope

Run a targeted validation limited to:

- task: `ap_001_idle_counter` only
- samples: exactly 3 samples
- profile: `neutral_baseline`
- model: configured/default `qwen36-27b` when available
- max tokens: unchanged `4096`
- temperature/top-p: unchanged deterministic defaults
- thinking disabled: use fixed runner path with configured model preset `extra_body`
- benchmark assets: use the clarified prompt and unchanged reference/testbench/workload/evaluator/scoring
- committed outputs: sanitized reports only

If endpoint or tools are unavailable, do not fake results. Emit sanitized blocker/failure rows or a sanitized note.

## Required Files

Create:

- `reports/v0.7_rfid_apbench_ap001_post_clarity_targeted_validation.md`
- `reports/v0.7_rfid_apbench_ap001_post_clarity_targeted_validation.csv`
- `reports/v0.7_rfid_apbench_ap001_post_clarity_targeted_validation.jsonl`

Do not overwrite historical reports.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- validation configuration
- confirmation that the clarified prompt was used
- confirmation that the fixed thinking-disabled runner path was used
- endpoint and tool status without secrets
- request outcome distribution
- per-sample sanitized request/content/token metadata
- extraction/evaluation gate summary
- correctness result summary
- valid-score/all-sample score summary if any rows pass
- comparison against the pre-clarity `ap_001` post-fix evidence
- interpretation of whether the clarity update improved `ap_001` candidate correctness
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next bounded direction
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Decision Rules

Recommend exactly one:

- `post_fix_expanded_10task_baseline_spec` if `ap_001` correctness improves or the remaining issue is no longer blocking a fixed-run expanded baseline.
- `ap001_clarity_closeout_spec` if `ap_001` remains failing but the clarity update is complete and should be documented without more immediate retries.
- `ap001_additional_analysis_spec` if the clarified prompt produces new or mixed failure modes needing classification.
- `benchmark_bugfix_spec` only if a concrete reference/testbench/asset bug is found.
- `no_immediate_followup_beyond_documentation` if validation cannot be performed or yields no useful evidence.

Do not recommend fine-tuning, private evaluation, model comparison, max-token increase, additional prompt-profile experiments, or training-data creation.

## Tests

Run:

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompt text
- raw model responses
- response bodies
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- add or modify benchmark tasks beyond the already clarified prompt
- edit reference RTL or testbench behavior
- change module interface
- run a benchmark-wide evaluation
- compare models
- change prompt profiles
- raise global max-token default
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change workload, metrics, area/activity scoring, or evaluator policy

## Definition of Done

Done when:

1. Sanitized Markdown/CSV/JSONL validation reports exist.
2. Exactly 3 `ap_001_idle_counter` samples are attempted, or endpoint/tool blockers are explicitly reported.
3. The report confirms the clarified prompt and thinking-disabled runner path were used.
4. Request/content/token metadata is reported without raw outputs.
5. Functional correctness outcomes are reported and compared against pre-clarity `ap_001` evidence.
6. Benchmark assets remain unchanged except for the already approved prompt clarity update.
7. Fine-tuning is not recommended from this evidence alone.
8. Exactly one next bounded direction is recommended.
9. `python -m pytest` passes.
10. No raw/private/model-output/training artifacts are committed.
