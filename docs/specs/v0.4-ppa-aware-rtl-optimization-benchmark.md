# Feature Spec: v0.4 PPA-Aware RTL Optimization Benchmark

## 1. Goal

Run the first small v0.4 public RTL-OPT control validation using the completed sanitized PPA export and reporting flow.

The v0.4 design docs, scoring core, reporting layer, and RTL-OPT result-row export layer are complete. The next step is a small end-to-end control validation, not a full benchmark and not a prompt comparison.

## 3. Next Stage

### Stage E: Small public RTL-OPT control validation

Status: next.

Run a small RTL-OPT validation slice using only:

- model: `qwen36-27b`
- prompt profile: `neutral_baseline`
- benchmark: `rtlopt`
- evaluator: Yosys equivalence/generic metric flow
- limit: 5 tasks
- samples per task: 1

This stage validates that the full flow works:

1. run a tiny public RTL-OPT slice
2. export structured `results.jsonl` rows to sanitized v0.4 PPA JSONL
3. summarize sanitized v0.4 PPA records to Markdown/CSV
4. commit only sanitized v0.4 artifacts and reports

## 4. Required Output Artifacts

Allowed committed artifacts:

- `artifacts/v0.4/ppa_control/rtlopt_neutral_baseline_ppa.jsonl`
- `reports/v0.4_ppa_control_summary.md`
- `reports/v0.4_ppa_control_summary.csv`

The JSONL artifact must contain only `PPAScoreRecord` fields. It must not contain raw response paths, extracted RTL paths, error log paths, prompts, model responses, generated RTL, source RTL, API keys, or private task text.

## 5. Validation Rules

Before committing, verify:

- the run is small, 5 RTL-OPT tasks only
- only `neutral_baseline` is used
- no held-back profile is run
- no full benchmark is run
- export uses `scripts/export_v0.4_ppa_results.py`
- summary uses `scripts/summarize_v0.4_ppa_results.py`
- committed JSONL loads through `load_sanitized_jsonl`
- summary separates scoring modes when applicable
- area-only results are labeled area-only
- no power improvement is claimed unless real power data exists

## 6. Review Questions

After the control summary is generated, answer:

1. Did the export/reporting flow work end to end?
2. How many samples were valid vs invalid?
3. Is the flow area-only or full area/power?
4. Are any metrics missing or invalid?
5. Is it safe to schedule a matched `neutral_baseline` vs `low_area_low_power` retest next?

## 7. Commit Rules

Allowed:

- sanitized v0.4 control JSONL artifact
- v0.4 control Markdown/CSV summaries
- documentation/spec updates only if needed
- test/helper fixes only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 8. Non-goals

Do not:

- run full benchmarks
- run `low_area_low_power` yet
- run `strict_rtl_only` yet
- add new models
- parse raw response files or extracted RTL files
- parse Yosys logs directly for the committed artifact
- integrate SiliconCraft private benchmarks
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim power improvement without measured power data
- claim final silicon PPA quality

## 9. Definition of Done for Next Codex Task

The next task is done when:

1. Small RTL-OPT `neutral_baseline` control run is complete.
2. Sanitized v0.4 PPA JSONL artifact exists.
3. v0.4 control Markdown summary exists.
4. v0.4 control CSV summary exists.
5. All committed artifacts contain no raw/private fields or raw-output paths.
6. `python -m pytest` passes.
7. No raw outputs are committed.
8. The final summary states whether the flow is ready for a matched `low_area_low_power` retest.
