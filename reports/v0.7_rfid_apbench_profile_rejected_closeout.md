# v0.7 RFID-APBench Prompt-Profile Experiment Rejected Closeout

## Scope

This report closes the bounded v0.7 prompt-profile experiment on the two public/synthetic RFID-APBench tasks with concentrated empty responses:

- `ap_001_idle_counter`
- `ap_010_retry_timeout_fsm`

The experiment compared the existing `neutral_baseline` control with one additive `completion_reliability` challenger. Each profile used `qwen36-27b` for exactly three samples per task, six samples per profile and 12 samples total, with identical deterministic generation settings. Benchmark task prompts, references, testbenches, workloads, metrics, extractor, evaluator, and scoring remained unchanged.

This closeout is documentation and planning only. It does not run another model evaluation, tune prompts, compare models, modify benchmark assets, create training data, perform private evaluation, or fine-tune a model.

## Profiles Compared

| role | profile | purpose |
| --- | --- | --- |
| control | `neutral_baseline` | Existing general RTL-generation system profile. |
| challenger | `completion_reliability` | Requests one concise, complete RTL module without prose, fences, hidden reasoning, or lengthy explanation and requires a final `endmodule` within the token budget. |

The challenger contains no task-specific answer, private example, private RTL, or training data.

## Experiment Outcome

### Request Outcomes

| profile | attempted | `success_empty` | `success_nonempty` | other request failures | empty rate |
| --- | ---: | ---: | ---: | ---: | ---: |
| `neutral_baseline` | 6 | 5 | 1 | 0 | 5/6 (83.3%) |
| `completion_reliability` | 6 | 5 | 1 | 0 | 5/6 (83.3%) |

Empty rates were also identical by task under both profiles:

| task | `neutral_baseline` | `completion_reliability` |
| --- | ---: | ---: |
| `ap_001_idle_counter` | 2/3 (66.7%) | 2/3 (66.7%) |
| `ap_010_retry_timeout_fsm` | 3/3 (100.0%) | 3/3 (100.0%) |

All 12 requests completed on the first attempt as parsed HTTP `2xx` responses. Every empty row had one response choice, null content, zero response characters, `finish_reason=length`, and 4096 completion tokens. No timeout, HTTP error, transport failure, retry, parse failure, or schema failure was observed.

### Evaluation Gates

| profile | generation | extraction | compile | correctness | synthesis | valid scores | all-sample score |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `neutral_baseline` | 6/6 | 1/6 | 1/6 | 0/6 | 1/6 | 0/6 | `0.000000` |
| `completion_reliability` | 6/6 | 1/6 | 1/6 | 0/6 | 1/6 | 0/6 | `0.000000` |

Under each profile, the sole non-empty response extracted, compiled, and synthesized but failed functional simulation. Compile and synthesis success are not functional correctness, and neither profile produced a valid score.

## Rejection Decision

Reject `completion_reliability` for further validation. It produced exactly the same aggregate empty rate, task-level empty concentration, request metadata, downstream gate counts, valid-score count, and failure distribution as the neutral control. It did not improve completion reliability or functional correctness.

No additional prompt-profile experiment is justified from this evidence alone. A different prompt profile would be speculative without new evidence about the repeated token-limit null-content behavior.

This rejection applies to this tested challenger and bounded evidence. It is not a claim that prompt profiles can never affect output, and it is not a comparison between models.

## Benchmark Asset Decision

Keep all benchmark task prompts and assets unchanged. Healthy reference validation and known-good candidate fixtures remain the authoritative evidence for benchmark integrity, while the matched profile outcomes identify no defect in the tasks, references, testbenches, workloads, extractor, evaluator, metrics, or scoring policy.

## Profile Inventory Decision

Leave `completion_reliability` defined but unused. Retaining the additive public profile preserves exact experiment reproducibility and avoids rewriting historical configuration, but it should not be selected for further validation or treated as recommended. Removing it would provide no behavioral benefit and would make the committed experiment less directly reproducible.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended from this evidence. The bounded public/synthetic experiment is not a training dataset and does not establish the volume, diversity, causal diagnosis, governance, or validation basis required for model-weight or adapter changes.

## Recommended Next Direction

Recommend exactly one bounded direction: `endpoint_runtime_investigation_spec`.

Both profiles repeatedly reached the 4096-token limit with parsed HTTP success, one choice, null content, and zero response characters. A future specification should investigate that sanitized serving/runtime boundary before more benchmark or prompt-profile work. It should not rerun a benchmark-wide baseline, compare models, change benchmark assets, expose raw response bodies, or create training data.

## Activity Proxy Caveat

Activity is a VCD toggle-count proxy from the declared public workload. It is not measured silicon power, signoff power, final silicon PPA, or production QoR. No activity score was available because no candidate passed functional correctness.

## Public Synthetic Caveat

This closeout covers only public/synthetic RFID-APBench evidence. It contains no private data or private RTL, is not private evaluation, and is not fine-tuning or training-readiness evidence.
