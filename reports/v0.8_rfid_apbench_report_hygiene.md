# v0.8 RFID-APBench Report Hygiene

## Scope And Non-Goals

This report documents the v0.8 RFID-APBench report hygiene guide created for committed public/synthetic benchmark evidence. The guide defines stable naming, required Markdown sections, per-sample CSV/JSONL schema expectations, sanitized metadata, evaluation gates, score summaries, source lineage, caveats, and prohibited artifact checks.

This is documentation/report-hygiene hardening only. No model evaluation, benchmark modification, prompt edit, reference RTL edit, testbench edit, model comparison, prompt-profile experiment, max-token change, private evaluation, training-data creation, fine-tuning, adapter work, or git tag creation was performed.

## Report Hygiene Document Created

Created:

- `docs/release/rfid_apbench_report_hygiene.md`

The document is review-facing and practical. It defines how new RFID-APBench evidence should be named, structured, sanitized, reviewed, and connected to prior reports before commit or release.

## Naming Convention Summary

The guide defines a stable report filename pattern:

```text
reports/v<major>.<minor>_rfid_apbench_<scope>[_<artifact>].md
reports/v<major>.<minor>_rfid_apbench_<scope>.csv
reports/v<major>.<minor>_rfid_apbench_<scope>.jsonl
```

It requires matching Markdown/CSV/JSONL stems for per-sample run artifacts and Markdown-only reports for planning, closeout, checklist, reproducibility, hygiene, and sanitized failure-analysis reports that do not produce new per-sample rows.

## Required Section Summary

The guide defines required Markdown sections for:

| report type | required coverage |
| --- | --- |
| baseline | scope, source evidence, run config, endpoint/tool status, request outcomes, gates, per-task summary, scores, interpretation, decisions, caveats |
| targeted validation | validation config, prompt/runner confirmations, per-sample metadata, gate/correctness/score summaries, comparison, interpretation, decisions, caveats |
| failure analysis | failure source summary, affected rows, expected behavior, sanitized taxonomy, defect support assessment, decisions, caveats |
| release closeout | timeline, inventory, final config, reliability status, baseline summary, known limitations, release readiness, tag example, next direction, caveats |
| planning/documentation | artifact summary, sections/workstreams covered, use with existing release docs, asset/fine-tuning decisions, next direction, caveats |

## Sanitized Row Schema Summary

For model-generation baselines and targeted validations, the guide requires row fields covering:

- sample identity and settings
- endpoint status
- request outcome and observability metadata
- content/token metadata
- generation and extraction status
- compile, functional simulation, synthesis, timing, area, and activity gates
- reference and generated area/activity metrics
- area, activity, and combined scores
- score validity
- failure category
- toolchain/workload IDs
- sanitized notes

For candidate-fixture evaluations, the guide defines a smaller row schema centered on candidate identity, gates, metrics, scores, failure category, toolchain/workload IDs, and notes.

Optional fields are allowed only when sanitized, stable, documented, and free of raw prompts, raw responses, generated RTL, logs, private paths, and secrets.

## Prohibited Artifact Summary

The hygiene document includes a pre-commit checklist excluding:

- private RTL and private task text
- raw prompts beyond public benchmark prompt text
- raw model responses and response bodies
- generated model RTL
- raw benchmark output directories
- VCD files or VCD snippets
- simulator logs and synthesis logs
- compiled artifacts and tool scratch
- endpoint credentials, secrets, and private absolute paths
- model weights, datasets, adapters, and fine-tuning scripts

## Use With Release Checklist And Reproducibility Docs

Use `docs/release/rfid_apbench_report_hygiene.md` with:

- `docs/release/rfid_apbench_release_checklist.md` for pre-tag review gates.
- `docs/release/rfid_apbench_reproducibility.md` for environment setup, deterministic settings, command shapes, output locations, and troubleshooting.

Recommended reviewer flow:

1. Use the reproducibility guide to understand how a report was generated and where raw artifacts should live.
2. Use the hygiene guide to check naming, sections, row schema, sanitized metadata, gates, scores, lineage, caveats, and prohibited artifacts.
3. Use the release checklist to verify the complete release evidence set before tagging.

## Benchmark Asset Decision

Benchmark assets remain unchanged by this report-hygiene documentation step. No task prompt, reference RTL, testbench behavior, module interface, workload, metric, scoring rule, evaluator policy, manifest identity, prompt profile, or max-token default was changed.

The only approved v0.7 benchmark asset change remains the previously completed `ap_001_idle_counter` public prompt clarity update. The `ap_006_wakeup_edge_filter` issue remains a candidate implementation limitation in the v0.7 evidence, not a benchmark asset defect.

## Fine-Tuning Readiness Decision

Fine-tuning remains out of scope and is not recommended. The report hygiene guide and v0.7/v0.8 release-hardening evidence are public/synthetic evaluation artifacts, not a governed training dataset or sufficient basis for model-weight changes, LoRA/QLoRA/DoRA adapters, or private-task adaptation.

## Recommended Next Bounded Direction

Recommend exactly one bounded next direction: `v0.8_ci_release_assertions_spec`.

Rationale: the release checklist, reproducibility guide, and report hygiene guide now define the manual release evidence gates. The next useful hardening step is to automate a bounded subset of those checks without requiring a live model endpoint.

## Activity Proxy Caveat

Activity is a VCD toggle-count proxy from the declared public workload. It is not measured silicon power, signoff power, final silicon PPA, or production QoR.

## Public Synthetic Caveat

This documentation covers only public/synthetic RFID-APBench evidence. It contains no private data or private RTL, is not private evaluation, does not create training data, and is not fine-tuning or training-readiness evidence.
