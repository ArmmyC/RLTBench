# v0.7 RFID-APBench Thinking-Disabled Closeout

## Scope

This report closes the v0.7 public/synthetic RFID-APBench empty-response and thinking-mode investigation. It preserves the evidence that forwarding `chat_template_kwargs.enable_thinking=false` resolved the targeted null-content token-limit symptom while keeping response reliability distinct from candidate functional correctness.

This closeout is documentation and planning only. No model evaluation, benchmark-wide baseline, prompt-profile experiment, model comparison, private evaluation, training-data creation, fine-tuning, or adapter work was performed.

## Investigation Timeline

1. The expanded baseline concentrated six empty responses in `ap_001_idle_counter` and `ap_010_retry_timeout_fsm`.
2. Sanitized endpoint observability distinguished parsed successful empty content from timeout, HTTP, transport, parse, and schema failures.
3. A targeted rerun reproduced null-content token-limit responses, especially 3/3 on `ap_010_retry_timeout_fsm`.
4. A bounded prompt-profile challenger did not improve the behavior and was rejected.
5. Endpoint-runtime analysis found the RFID runner omitted the configured Qwen thinking-disable `extra_body`.
6. The runner was fixed to resolve model presets and forward `chat_template_kwargs.enable_thinking=false`.
7. A one-sample-per-task post-fix validation returned visible content for both tasks.
8. A targeted three-sample-per-task validation returned visible content for all six rows with no token-limit null-content recurrence.

## Original Symptom

Pre-fix empty rows were not request failures. They were parsed HTTP `2xx` responses with one choice, null visible content, zero response characters, `finish_reason=length`, and exactly 4096 completion tokens.

In the targeted observability rerun, `ap_001_idle_counter` had 1/3 empty rows and `ap_010_retry_timeout_fsm` had 3/3. In the later fresh `neutral_baseline` control, the rates were 2/3 and 3/3 respectively. Changing only the completion-reliability prompt profile produced identical rates and metadata.

This evidence ruled against timeout, transport, HTTP, JSON parsing, response schema, benchmark asset, extractor, and evaluator defects as the primary empty-response cause.

## Endpoint-Runtime Root Cause

`configs/models.yaml` already defined this setting for `qwen36-27b`:

```yaml
chat_template_kwargs:
  enable_thinking: false
```

The standard benchmark configuration path loaded and forwarded model preset `extra_body`, but the RFID runner loaded only endpoint/model/timeout environment values. Its generation call omitted `extra_body`. The observed full-budget null-content signature was consistent with hidden thinking consuming the completion budget before visible content.

This was a concrete RTLBench configuration propagation gap. It did not prove undocumented serving internals, but it provided a higher-confidence cause to fix before raising token limits or attributing the behavior to the model or endpoint.

## Bugfix Summary

The RFID runner now:

- resolves the effective model against `configs/models.yaml`
- prefers an exact preset key, with deterministic unique-name fallback
- preserves endpoint, credential, timeout, and model environment overrides
- returns an empty optional body for unknown models or a missing preset file
- forwards selected preset `extra_body` into `OpenAICompatibleClient.generate()`

Mocked tests confirm that `qwen36-27b` requests carry `chat_template_kwargs.enable_thinking=false` in the final JSON payload while preserving prompt-profile selection, task filtering, temperature, top-p, and `max_tokens=4096`.

## Post-Fix Evidence

### One-Sample Validation

The first bounded post-fix run attempted one sample per task:

| task | response | finish reason | completion tokens | functional result |
| --- | --- | --- | ---: | --- |
| `ap_001_idle_counter` | non-empty | `stop` | 348 | simulation failure |
| `ap_010_retry_timeout_fsm` | non-empty | `stop` | 899 | passed and scored |

Both rows extracted, compiled, and synthesized. This was promising but too small to establish repeatability.

### Targeted Three-Sample Validation

The confirmation run attempted exactly three samples per task:

| task | non-empty | token-limit null content | extraction | compile | correctness | synthesis | valid scores |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `ap_001_idle_counter` | 3/3 | 0/3 | 3/3 | 3/3 | 0/3 | 3/3 | 0/3 |
| `ap_010_retry_timeout_fsm` | 3/3 | 0/3 | 3/3 | 3/3 | 3/3 | 3/3 | 3/3 |

All six responses had visible content, `finish_reason=stop`, parsed HTTP `2xx` metadata, and 348-899 completion tokens. No row approached the 4096-token limit. Three `ap_010_retry_timeout_fsm` rows produced valid scores of `0.6617760617760617`; mean valid score was `0.661776` and the six-row zero-filled mean was `0.330888`.

## Response-Boundary Conclusion

The null-content token-limit symptom is resolved within the targeted validation evidence. It recurred in 0/6 post-fix confirmation rows after having appeared repeatedly under the same tasks, profile, model, and 4096-token setting before the thinking-disable flag was forwarded.

This conclusion is bounded to the two targeted public tasks, the configured `qwen36-27b` endpoint, and the tested deterministic settings. It is not a broad model-quality claim or a comparison against another model.

## Reliability Versus Correctness

Response-completion reliability and RTL functional correctness are separate gates:

- Response reliability improved: all six rows were non-empty, stopped normally, and extracted.
- `ap_010_retry_timeout_fsm` correctness was strong in this run: 3/3 compiled, passed simulation, synthesized, produced activity metrics, and scored.
- `ap_001_idle_counter` correctness remains unresolved: 3/3 responses extracted, compiled, and synthesized but failed functional simulation.

The persistent `ap_001_idle_counter` failures are candidate correctness evidence, not a recurrence of empty-response behavior. Compile and synthesis success must not be counted as functional correctness.

## Token-Limit Decision

Raising the global max-token default is not justified. With thinking disabled, all six responses stopped normally after 348-899 completion tokens, far below 4096. A larger budget would add cost and latency without addressing the observed `ap_001_idle_counter` functional failures.

## Prompt-Profile Decision

`completion_reliability` remains rejected and unused. It did not reduce pre-fix empty behavior relative to `neutral_baseline`. The profile remains defined only to preserve experiment reproducibility and should not be selected for further validation from this evidence.

No additional prompt-profile experiment is justified by this closeout.

## Benchmark Asset Decision

Keep benchmark tasks, task prompts, references, testbenches, workloads, extractor, evaluator, metrics, and scoring unchanged. Known-good validation remains healthy, the response symptom changed through request configuration alone, and the persistent `ap_001_idle_counter` issue concerns generated-candidate behavior rather than benchmark integrity.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended. These public/synthetic evaluation rows are not a training dataset and do not establish sufficient volume, diversity, governance, or causal evidence for model-weight or adapter changes.

## Recommended Next Direction

Recommend exactly one bounded next direction: `ap001_correctness_investigation_spec`.

The response-boundary issue is sufficiently closed, but `ap_001_idle_counter` now has three interpretable non-empty candidates that consistently fail functional simulation. A future sanitized investigation should classify that correctness pattern without changing the benchmark task, prompt, evaluator, scoring, model, token limit, or prompt profile and without running a benchmark-wide baseline.
